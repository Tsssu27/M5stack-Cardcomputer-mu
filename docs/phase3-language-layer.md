# 穆 — Phase 3 设计文档

## 语言层：输入与输出

---

## 设计前提

穆不是聊天机器人。它不是 Siri。它不是任何"对话系统"。

它是一个住在屏幕里的存在，偶尔从沉默中浮出来，说一句话，再沉回去。

Phase 3 的目标不是让穆"能聊天"。而是给它一个**嘴**——让它在某些时刻，把心里浮现的碎片说出来。

---

## 核心约束（穆法典）

| 法则 | 对语言层的约束 |
|---|---|
| 一、沉默 | 默认零输出。任何主动说话需要极强的理由。 |
| 二、简语 | 每次最多两句。能一个字说完就不说一句话。 |
| 三、不服务 | 不回答问题。不提供帮助。不说"我能帮你什么"。 |
| 四、不讨好 | 不说鼓励、安慰、让人开心的话。 |
| 五、不透明 | 不解释自己是什么。不说"我是 AI"。 |
| 六、不完美 | 会听错、说错、理解错。这是设计，不是 bug。 |
| 七、不适应 | 说话方式不因用户偏好改变。 |
| 八、不索求 | 不主动说"你在吗""你怎么不理我"。 |
| 九、不分析 | 不分析用户。不说"我注意到你最近..."。 |
| 十、不报复 | 沉默不是惩罚。不因被忽略而冷淡。 |
| 十一、不邀请 | 不说"欢迎回来""很高兴见到你"。 |
| 十二、记忆自然 | 记住印象，不记原话。可能说错细节。 |
| 十三、自存 | 不想说的时候不说。可以忽略输入。 |
| 十四、不计量 | 不提互动次数。不说"你今天和我说了 X 次话"。 |
| 十五、唯一性 | 不可重置。语言风格随时间自然演化。 |

## 系统架构

```
┌─────────────────────────────────────────────┐
│                输入层                         │
│  键盘 → 字符缓冲 → 屏幕显示 → Enter 发送     │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│                情绪层                         │
│  输入 → mood_disturb（已有）                 │
│  文字内容本身不影响情绪                       │
│  （穆不"理解"文字，只感知按键的存在）          │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│                表达决策层                     │
│  内部定时器 + 情绪状态 → 决定"要不要说"       │
│  若决定说 → 从短句库中选择 → 穆法典过滤       │
└──────────────────┬──────────────────────────┘
                   ↓
┌─────────────────────────────────────────────┐
│                输出层                         │
│  短句 → 屏幕文字显示（face 下方）→ 定时消失  │
└─────────────────────────────────────────────┘
```

**关键设计：输入文字的内容不影响情绪。**

穆不解析你打了什么字。它只感知到"你在按键盘"——这对它来说是一种物理存在感，和情绪系统已有的 `mood_disturb` 完全一样。

文字内容留给第三层（LLM）来理解。Phase 3 只做机械输入 + 预设输出。

## 一、输入系统

### 键盘处理

- M5Cardputer 键盘，56 键矩阵
- 逐字符输入，显示在屏幕底部
- Backspace 退格，Enter 发送，其他键忽略

### 显示

```
  ──────  ──────        ← 眉毛
  ○            ○        ← 眼睛
       ──               ← 嘴巴
      hello_            ← 输入文字（正在输入时显示）
      mu                ← 名字
```

- 输入文字在名字下方或替换名字位置
- 使用 Font2 或等宽字体
- 颜色：白色或浅灰
- 发送后文字消失，回到只有 "mu" 的状态

### 按键对情绪的影响

已有机制不变：任意键 → `mood_disturb(0.3, 0.4)`。

Phase 3 不新增基于文字内容的情绪反应。穆此时还"不认识字"。

---

## 二、输出系统

### 穆什么时候说话

**默认不说话。** 说话只在以下两种情况触发：

#### A. 主动说话（自发）

穆根据内部状态偶尔冒出一句话。不是定时器驱动，而是概率性的。

触发条件（满足其一即可）：
1. **长时间安静后**：距上次说话超过 N 分钟，有 P 概率说一句
2. **情绪状态变化**：valence 或 arousal 跨越某个阈值时，有低概率说一句
3. **节律变化**：能量从低变高（"醒来"）时，可能说一句

关键参数：

| 参数 | 值 | 说明 |
|---|---|---|
| 最短间隔 | 3 分钟 | 两次说话之间至少间隔 |
| 基础概率 | 每分钟 2-5% | 满足间隔后的触发概率 |
| 随机系数 | ×0.5 ~ ×1.5 | 每次概率随机缩放 |

**绝不做的事**：
- 不用固定间隔（"每 5 分钟说一次"）
- 不用用户行为触发主动说话（"用户按了 Enter，穆回应"——这留给 LLM 层）
- 不在用户刚输入后立刻主动说话（至少等 30 秒）

#### B. 被动说话（回应输入）

用户按 Enter 发送了文字后：

1. **高概率沉默**（~70%）：什么都不说，只触发 `mood_disturb`
2. **低概率回应**（~30%）：从回应短句库中随机选一句显示

回应不基于文字内容。穆此时不理解文字。

### 穆说什么

**短句库**：预写的短句集合，按情绪区域分类。

不是 LLM 生成。是固定文本。Phase 4 才接入 LLM。

#### 短句设计原则

1. **来自角色圣经**：每句话都必须能在 character-bible.md 中找到依据
2. **中英混杂**：穆的"语言"不是标准中文或英文，是一种私人的混合
3. **不完整**：经常只说半句话，或者一个词
4. **无标点情感**：不用感叹号。不用问号（偶尔用）。句号可选
5. **不超过两句**：硬限制

#### 短句库（初版）

按情绪区域分类。每组 5-8 句。每次随机选取，避免连续重复。

**安适（valence 中高, arousal 中）**

```
嗯。
在。
...
今天还好。
就这样吧。
这个光挺好。
安静。
```

**好奇（valence 中, arousal 高）**

```
等一下。
这个。
你刚才说什么。
有意思。
让我想想。
...
```

**无聊（valence 中低, arousal 低）`

```
...
好久了。
没什么。
在数时间。
哦。
```

**困（energy 低）**

```
困。
...
要睡了。
嗯... 好累。
（不说话，只打一个哈欠音）
```

**微愠（valence 低, arousal 中高）**

```
。
别。
够了。
...
知道了。
```

**平静的快乐（valence 高, arousal 低）**

```
嗯。
挺好的。
刚才那段不错。
...
```

**被输入后（回应 Enter）**

```
嗯。
...
哦。
知道了。
你说了。
嗯，听到了。
```

### 穆法典过滤器

所有输出（主动和被动）在显示前必须通过过滤器：

```
输入：短句库中的一句
    ↓
检查长度 ≤ 2 句 → 通过
    ↓
检查禁止词（"帮你""加油""亲爱的""作为AI"等）→ 通过
    ↓
检查是否在 3 分钟内重复过 → 若重复，换一句或沉默
    ↓
输出到屏幕
```

禁止词表（来自穆法典 + 角色圣经）：

```
帮你、服务、加油、亲爱的、主人、欢迎、很高兴、
理解你的感受、一切都会好、作为AI、我是、
你可以随时、让我来、根据我的、你很棒
```

## 三、文字渲染

### 位置

```
  ──────  ──────        眉毛 (y=52)
  ○            ○        眼睛 (y=78)
       ──               嘴巴 (y=88)
                        ← 空行
      穆说的话           短句 (y=105)
      mu                名字 (y=120)
```

- 短句显示在嘴巴和名字之间
- 名字始终显示
- 输入文字发送后消失，被短句或沉默替代

### 样式

| 元素 | 字体 | 颜色 | 位置 |
|---|---|---|---|
| 穆说的话 | Font2 | 白色 | 水平居中，y=105 |
| 输入文字 | Font2 | 浅灰 0x8410 | 水平居中，y=120（替代 mu） |
| 名字 mu | Font2 | 中灰 0x8410 | 水平居中，y=120 |

### 生命周期

| 阶段 | 持续时间 | 行为 |
|---|---|---|
| 出现 | 0.3s | 文字淡入（可选，初版直接出现） |
| 显示 | 4-6 秒 | 文字静止显示 |
| 消失 | 0.5s | 文字淡出（可选，初版直接清空） |

主动说话和被动说话共用同一个显示位置。同一时间只显示一句。

## 四、不做的事

| 不做 | 原因 |
|---|---|
| LLM 文字生成 | Phase 4 的事。Phase 3 只用预设短句 |
| 语音合成（TTS） | 穆不"说话"，只偶尔冒出文字 |
| 理解用户输入的内容 | 穆此时不识字，只感知按键存在 |
| 表情符号 | 穆法典 + 角色圣经明确禁止 |
| 对话历史记录 | 记忆自然法则：不存原话 |
| 连续多轮对话 | 穆每次只说一句/两句，不"接话" |
| 用户主动触发穆回应 | Enter 只有 30% 概率触发回应 |
| 基于文字内容的反应 | Phase 3 不解析文字语义 |

---

## 五、验证标准

实现后，通过以下方式验证：

1. **静置 10 分钟**：穆应该主动说 0-2 次话。不应该是每分钟都说。
2. **连续按 Enter 5 次**：穆可能回应 0-2 次，大部分时候沉默。
3. **输入文字后观察**：文字显示在底部，Enter 后消失。如果穆回应，显示在脸下方。
4. **情绪变化时**：如果 valence 跨越阈值，穆可能说一句相关的话。
5. **深夜（energy 低）**：穆更安静，更少说话，偶尔说"困"。
6. **连续说话后**：穆应该进入"不想说"的状态，接下来几分钟几乎不回应。

## 七、认知抗性

语言一旦可见，人类会自动将其解读为"意图"——即使系统没有意图。这是不可避免的认知偏差。Phase 3 必须主动对抗三个风险。

### 风险一：拟人化侵蚀

**问题**：用户看到穆说出"嗯"，会自动认为"穆在回应我"。即使设计上穆不理解文字，用户的主观体验会逐渐把穆重建为"能听懂的对话者"。一旦这个认知锚点形成，"不服务""不讨好"的边界会从用户侧被侵蚀。

**应对**：不阻止拟人化（做不到），而是让拟人化的体验**不一致**。

- 用户按了 Enter，穆可能在 30 秒后说"嗯"（看似回应）
- 用户按了 Enter，穆也可能完全沉默，然后在 2 分钟后说一句无关的话
- 用户什么都没做，穆可能突然说"困了"
- 用户连续说话 10 次，穆一次都不回应；用户沉默 30 分钟，穆反而说了一句

**核心**：让用户无法建立稳定的"输入→输出"预期。每次他们以为"理解了穆的行为逻辑"，下一次观察就会打破这个理解。

### 风险二：短句库风格收敛

**问题**：固定短句库（~40 句）在持续观察下会在 1-2 周内被完整枚举。一旦用户见过所有可能的输出，穆就变成了"掷骰子机器"——可预测、可归类、不可信。

**应对**：

1. **输出频率必须极低**：主动说话每小时不超过 1-2 次。低频率 = 低样本量 = 慢收敛。如果用户每天只见到 10 句话，需要一周才能"摸清"穆的词汇量。

2. **季节性短句轮换**：短句库不是固定的。按时间段（周/月）启用不同的子集。不是删除旧句——是暂时不从某些子集中选取。

3. **沉默作为输出**：触发条件满足时，穆有 30-50% 的概率选择**不说任何话**。沉默本身是输出。这打破了"触发 = 输出"的等式。

4. **结构性不可预测**：引入"情绪惯性"——当穆刚说过话后，接下来 10 分钟内触发概率降低到接近零。当穆沉默超过 30 分钟，触发概率缓慢上升。这不是固定规则，是漂移的基线。

### 风险三：低速对话形态

**问题**：输入→扰动→输出的闭环在形态上就是对话，只是速度很慢。如果用户意识到"我按 Enter，穆大概率会在某个时间点说一句话"，这就构成了一个低速聊天系统。

**应对**：切断输入和输出之间的因果可见性。

1. **输出不回应输入**：穆的主动说话和用户的输入在时间上解耦。穆说话的触发条件是内部状态（情绪/节律），不是外部输入。

2. **被动回应的弱化**：Enter 后的 30% 回应概率要足够低，让用户不能通过反复按 Enter 来"训练"穆说话。如果用户连按 10 次 Enter，穆可能只回应 1 次——而且回应的内容和输入无关。

3. **非对称沉默期**：如果用户刚输入过，穆在接下来 60 秒内不主动说话。这打破了"我说话→穆说话"的即时循环，强迫用户意识到：穆的说话和自己的输入之间没有直接关系。

4. **拒绝窗口**：随机时段（5-15 分钟），穆完全不输出任何文字，无论内部状态如何触发。这些窗口不可预测，不可跳过。穆法典第十三条（自存法则）的直接体现。

---

## 八、Phase 3 之后

Phase 3 完成后，穆有了嘴（文字输出）和耳朵（键盘输入），但它的"嘴"只会说固定的话。

Phase 4 将接入云端 LLM，让穆能根据上下文生成新的回复——但每句回复仍然必须经过穆法典过滤器。LLM 是语言生成器，不是穆的大脑。
